This is the comprehensive architectural blueprint for your "Mobile AI IDE." We
are moving away from "how to build it" (tech stack) and focusing entirely on
"how it works" (the logic and flow).

Core Architecture: The Intelligence Loop

The system is designed as a triangular relationship between the User Interface,
the Orchestration Brain, and the Execution Sandbox.

1. The High-Level Structural Components

A. The Control Center (User Interface)

The UI is not just a "skin"; it is the Configuration Layer. Its primary jobs
are:

  - Identity Management: Holding the vault of multiple Gemini accounts/keys.
  - Agent Switching: Defining which "Persona" (Agent) is currently active.
  - View Synchronization: Rendering the code editor and the terminal in
    real-time based on what is happening in the sandbox.

B. The Orchestrator (The Brain)

The Orchestrator is the middleware that manages the conversation. It acts as the
Translator between the user's intent, the AI's reasoning, and the sandbox's
capabilities. It handles:

  - Context Assembly: Gathering the right files, terminal logs, and agent
    instructions to send to Gemini.
  - Tool Dispatching: Recognizing when Gemini wants to "do" something (like run
    a command) and sending that command to the sandbox.
  - Identity Routing: Ensuring the request is sent to the specific Gemini
    account assigned to that agent.

C. The Execution Sandbox (The Environment)

The Sandbox is a Isolated State. It is a living environment where the actual
code lives. It provides:

  - The File System: A place to create, read, and edit files.
  - The Shell: A command-line interface that can execute scripts, install
    packages, and run tests.
  - The Event Stream: A constant feed of output (STDOUT/STDERR) that can be fed
    back to the AI.

2. The Core Logic Flows

Flow I: The "Identity & Agent" Handshake

Before any code is written, the system must establish Who is talking and Which
AI is thinking.

1.  Identity Assignment: The user selects a "Gemini Identity" (Account A) and
    assigns it to an "Agent" (e.g., The Backend Architect).
2.  Agent Definition: The Agent is defined by a System Prompt (instructions on
    how to behave) and a Permission Set (what functions it is allowed to call in
    the sandbox).
3.  Session Binding: When the user opens a project, the Orchestrator binds the
    specific Agent and Identity to that session.

Flow II: The "AI-to-Action" Loop (The Tool Use)

This is the heart of the "Cursor" experience. It is a recursive loop:

1.  The Request: User asks: "Fix the bug in the login logic."
2.  Context Gathering: The Orchestrator looks at the open file and the last 50
    lines of the terminal. It packages this with the Agent's instructions.
3.  The AI Reasoning: Gemini receives the package. Instead of just replying with
    text, it replies with a Tool Call (e.g., read_file("auth.js")).
4.  The Interception: The Orchestrator sees the tool call, pauses the chat, and
    sends a request to the Sandbox to read that file.
5.  The Execution: The Sandbox reads the file and sends the content back to the
    Orchestrator.
6.  The Feedback: The Orchestrator feeds the file content back to Gemini.
7.  The Final Action: Gemini now has the code \rightarrow identifies the bug
    \rightarrow issues another Tool Call: write_file("auth.js",
    "fixed_code_here").
8.  The Reflection: The Sandbox updates the file \rightarrow the UI refreshes
    the editor \rightarrow the user sees the change happen in real-time.

Flow III: The CLI Session Integration

Unlike a standard chat, the CLI is a bidirectional stream.

1.  User \rightarrow CLI: The user types a command manually; the Sandbox
    executes it.
2.  CLI \rightarrow AI: If the command fails (returns an Error), the
    Orchestrator automatically captures that error and asks the active Agent:
    "The user ran this command and it failed with this error. How do we fix it?"
3.  AI \rightarrow CLI: The Agent suggests a fix and, with user permission,
    executes the fix directly in the terminal.

3. Summary of the "State" Management

To make this feel seamless on a phone, the system manages three different
"States":

| State Type          | Where it lives | What it stores                                                                            |
| :------------------ | :------------- | :---------------------------------------------------------------------------------------- |
| **Auth State**      | User Interface | Multiple Google Tokens / API Keys.                                                        |
| **Context State**   | Orchestrator   | Current conversation history, active agent prompt, and "memory" of the project structure. |
| **Execution State** | Sandbox        | The actual files, installed dependencies, and the running process of the CLI.             |

[ User ] \rightarrow (Selects Agent/Identity) \rightarrow [ UI ]
\downarrow \uparrow [ Orchestrator ] \leftarrow (Sends Context) \rightarrow [
Gemini (Identity A/B/C) ] \downarrow \uparrow [ Sandbox ] \leftarrow (Executes
Functions: Read/Write/Run) \rightarrow [ Files/Shell ]
